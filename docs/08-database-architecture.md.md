# Database Architecture Specification

Version: 1.0

Status: Approved

---

# Purpose

This document defines the database architecture principles for the Headless SEO CMS Platform.

It does not describe individual tables.

Instead, it defines how the database must be designed.

Every schema, migration, index and relationship must follow these rules.

The primary objective is to create a database that remains maintainable, scalable and performant for many years.

---

# Database Engine

The application must use

PostgreSQL

hosted by

Supabase.

PostgreSQL features should be preferred whenever they provide measurable advantages over generic SQL implementations.

---

# ORM

Prisma ORM is mandatory.

All communication with PostgreSQL must happen through Prisma.

Direct SQL queries are forbidden except when PostgreSQL-specific optimizations cannot be achieved through Prisma.

---

# Database Philosophy

The database is not merely a storage layer.

It is one of the core architectural components of the application.

The schema must express business relationships clearly.

The database should help prevent invalid application states rather than relying solely on application logic.

---

# Design Principles

The schema must satisfy:

• Normalization

• Readability

• Predictability

• Strong Relationships

• Explicit Constraints

• Minimal Duplication

• Query Performance

• Future Scalability

---

Primary Keys

Every table uses

UUID

generated by PostgreSQL.

Never use incremental integer IDs.

Reasons:

- globally unique
- safer public identifiers
- easier replication
- future multi-tenant support

---

# Timestamp Strategy

Every mutable entity must contain

created_at

updated_at

Optional

deleted_at

Never overwrite creation timestamps.

updated_at must update automatically.

---

# Audit Strategy

Every important mutation must be traceable.

The system must record

Who

What

When

Previous value

New value

Operation

Target entity

Audit records must never be modified.

---

# Soft Delete Strategy

Entities should support soft deletion where business value exists.

Examples

Products

Categories

Pages

Media

Redirects

Settings

Soft deleted entities remain recoverable.

Permanent deletion is a privileged operation executed only after confirmation.

---

# Hard Delete

Use only when recovery provides no business value.

Examples

Temporary caches

Expired sessions

Generated previews

Transient analysis data

---

# UUID Strategy

Every public entity uses UUID.

Never expose internal database identifiers.

URLs must use slugs rather than UUIDs whenever possible.

---

# Relationship Strategy

Prefer explicit foreign keys.

Never store references inside text fields.

Relationships must be enforced by the database.

---

# Cardinality

One-to-One

One-to-Many

Self-Referencing

# Cascading Rules

# File Storage Strategy

Binary files must never be stored in PostgreSQL.

Only metadata belongs in the database.

Actual files belong to Supabase Storage.

Database stores:

Path

Filename

Dimensions

Size

Mime Type

Hash

ALT text

Created Date

---

# Language Strategy

The platform supports

Ukrainian

English

Multilingual content must be modeled explicitly.

Every translatable field must exist independently for each language.

Avoid storing multilingual content inside opaque JSON blobs.

The schema should remain queryable.

---

# SEO Strategy

SEO is treated as a first-class domain.

SEO data must not be mixed with product business data.

SEO information belongs to dedicated entities.

Benefits

Independent evolution

Versioning

Future APIs

SEO templates

SEO analysis

---

# Versioning Strategy

Every significant business entity supports version history.

Version records are immutable.

Restore operations create new versions rather than modifying old ones.

---

# Migration Strategy

Prisma migrations only.

Never edit migration history manually.

# Backup Strategy

Database backups handled by Supabase.

Application architecture must assume recovery from backups is possible.

Never depend on backups for normal undo functionality.

---

# Security

Every query must execute through authenticated services.

Never expose unrestricted table access.

Apply Row Level Security where appropriate.

Sanitize all user input before persistence.

Validate all data before writing.

---

# Database Quality Checklist

Every table must answer the following questions before creation:

- Does it represent a real business entity?
- Does it have a clear owner?
- Does it duplicate existing data?
- Are relationships explicit?
- Are indexes justified?
- Are constraints complete?
- Does it support future growth?
- Is the naming consistent?
- Can it be queried efficiently?
- Does it respect audit requirements?

If any answer is "No", the schema must be redesigned before implementation.
